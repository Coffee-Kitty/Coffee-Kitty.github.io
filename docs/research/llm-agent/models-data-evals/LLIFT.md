>## 📖 故事线：问题 → 挑战 → 解决方案
>
>### 第一幕：问题是什么？
>
>#### UBI漏洞（Use Before Initialization）
>
>```c
>// 危险的代码
>int x;           // 声明变量
>func(&x);        // 可能初始化，也可能不初始化
>use(x);          // 直接使用 ← 如果没初始化就危险了！
>```
>
>
>
>**为什么危险？**
>
>- 未初始化的变量包含随机值
>- 可能导致程序崩溃、安全漏洞
>- 在Linux内核中尤其严重
>
>------
>
>### 第二幕：现有工具的困境
>
>#### UBITect（已有的静态分析工具）
>
>```
>第一步：静态分析 → 找到14万个"可能有问题"的地方
>         ↓
>第二步：符号执行 → 验证哪些是真的有问题
>         ↓
>结果：60%成功验证
>      40%超时/内存不够 ← 这些案例怎么办？❓
>```
>
>
>
>**为什么会超时？**
>
>看一个例子：
>
>```c
>int ret = sscanf(str, "%u.%u.%u.%u", &a, &b, &c, &d);
>if (ret >= 4) {
>    use(a, b, c, d);
>}
>```
>
>
>
>**符号执行的困难**：
>
>- 需要分析sscanf内部（有复杂的循环）
>- 需要枚举str的所有可能值
>- 路径太多 → 10分钟还分析不完 → 放弃 ❌
>
>------
>
>### 第三幕：LLift的解决方案
>
>#### 核心想法：让GPT-4来帮忙
>
>**不需要精确分析sscanf内部，直接问GPT-4：**
>
>```
>问题：
>"当sscanf返回值 >= 4时，变量a, b, c, d是否一定被初始化了？"
>
>GPT-4回答：
>"是的！返回值表示成功解析的参数个数。
> 如果 >= 4，说明至少解析了4个参数，
> 所以a, b, c, d一定都被初始化了。"
>
>结论：这不是漏洞 ✓
>```
>
>
>
>**为什么GPT-4能做到？**
>
>- 它在训练时见过大量代码
>- "理解"了sscanf的语义
>- 不需要逐行执行代码
>
>------
>
>## 🔧 LLift的四个关键技巧
>
>### 技巧1：后置约束引导（最重要！）
>
>**简单说**：从"使用的地方"往回看，而不是从"所有可能的地方"往前看
>
>```c
>// 从这里往回看
>if (ret >= 4) {  // ← 知道了这个条件
>    use(a, b, c, d);
>}
>
>// 问题变成：
>// "在ret >= 4的情况下，a,b,c,d是否初始化？"
>// 而不是：
>// "在所有可能的情况下，a,b,c,d是否初始化？"
>```
>
>
>
>**好处**：问题范围大大缩小 ✂️
>
>------
>
>### 技巧2：渐进式提示
>
>**简单说**：不一次性给GPT-4所有代码，而是它需要什么再给什么
>
>```
>LLift: "分析这个函数"
>GPT-4: "我不认识这个函数，能给我看看定义吗？"
>LLift: "好，这是定义..."
>GPT-4: "这里面又调用了另一个函数..."
>LLift: "好，这是那个函数..."
>...
>```
>
>
>
>**好处**：
>
>- 节省token（GPT-4有字数限制）
>- 避免一次性信息过载
>
>------
>
>### 技巧3：任务分解
>
>**简单说**：把复杂问题拆成两步
>
>```
>对话1（找线索）：
>Q: "这个变量的初始化器是什么？"
>A: "是sscanf函数"
>Q: "使用它的条件是什么？"
>A: "ret >= 4"
>
>对话2（分析）：
>Q: "当ret >= 4时，sscanf是否保证初始化变量？"
>A: "是的，必定初始化"
>```
>
>
>
>**好处**：
>
>- 每步任务简单明确
>- GPT-4不容易混淆
>
>------
>
>### 技巧4：自我验证
>
>**简单说**：让GPT-4检查自己的答案
>
>```
>GPT-4第一次回答: "可能未初始化"
>
>LLift: "请检查：
>       1. 是否考虑了后置约束？
>       2. 当ret >= 4时是否保证初始化？
>       请重新审查你的答案。"
>
>GPT-4修正后: "对不起，我重新看了，
>             当ret >= 4时确实保证初始化。
>             答案应该是：必定初始化"
>```
>
>
>
>**好处**：减少GPT-4的错误
>

>### 1. **后置约束引导路径分析**
>
>- **传统**：分析所有可能的执行路径 → 路径爆炸

## Abstract

​	静态分析是软件工程中广泛应用的一种技术，用于发现和减轻程序中的缺陷。然而，实现**精确性与可扩展性之间的平衡**一直是一个重大挑战。大型语言模型（LLM）为此提供了一个有前景的替代方案——近年来的研究表明，它们在理解、生成甚至调试代码方面展现出卓越的能力。然而，`缺陷的逻辑往往复杂，常常需要跨越多个函数范围的**复杂推理与大规模分析能力**。`因此，目前 LLM 更适合以**辅助角色**来补充静态分析。

​	本文深入探讨了 **LLM 辅助静态分析（LLM-assisted static analysis）** 的研究空间，并以 **未初始化前使用（Use-Before-Initialization, UBI）** 缺陷为案例研究。为此，我们提出了 **`LLift`**——一个可完全自动化的框架，能够同时与静态分析工具和 LLM 交互。通过对框架结构与提示设计（prompt design）的精心构思，我们克服了多个挑战，<u>**包括缺陷特定建模、问题范围过大以及 LLM 的非确定性**</u>等。

​	在对近千个由静态分析检测出的潜在 UBI 缺陷进行实测时，LLift 展现出强大的能力，取得了**合理的精确率（50%）**，并且**未出现漏检缺陷**。此外，**`LLift 还在 Linux 内核中发现了 13 个此前未知的 UBI 缺陷`**。本研究为在大规模真实场景中利用 LLM 进行缺陷发现开辟了新的研究方向与方法论



## Intro



​	静态分析是软件工程中广泛应用的一项关键技术，尤其在缺陷检测领域，能够有效提升代码的质量、可靠性与安全性。**`然而，这类技术的有效性受到精确性与可扩展性之间的根本权衡所限制，尤其是在处理大型且复杂的程序时 [9, 24]。一方面，低精度的静态分析方法往往会产生大量误报（false positives）；另一方面，具有高精度的昂贵静态分析或符号执行方法则常常难以在合理时间内完成分析。`**因此，要在如 **Linux 内核** 这样规模庞大的程序上实现**全面且精确的静态分析**，依然是一个重大的挑战。

​	 `UBITect` [40] 是一个功能强大的静态分析方案，充分揭示了静态分析中固有的局限性。该工具针对 **Linux 内核中的 Use-Before-Initialization（UBI）** 错误，构建了一条分析管线：(1) 采用**bottom to up的摘要式静态分析**以实现可扩展性，但精度有限；(2) 结合**高精度的符号执行**以提高分析准确性，但其可扩展性受限。该方法清楚地表明，要在精度与可扩展性之间取得平衡，亟需新的替代策略。尽管这种双阶段的分析思路具有策略性，**`但在实践中，静态分析阶段报告的潜在漏洞中，近 40% 的案例在后续的符号执行阶段因超时或内存耗尽而无法得到结论性结果`。**这一限制削弱了工具的整体有效性，使得分析面临两种不理想的结果：若忽略这些报告，可能**漏报漏洞**（UBITect 采用的做法）；若交由开发者检查，则可能导致**误报（false positives）**。

​	在本文中，我们探讨了利用**大语言模型（LLMs）**来处理此类“疑难案例”的可能性。这一思路的提出，源于近年来的研究表明，LLMs 在代码理解、生成乃至调试方面展现出了强大的潜力 [4, 8, 13]。然而，将 LLM 应用于漏洞发现领域依然充满挑战。GPT-4 的技术报告对此已有明确指出：**在发现新型漏洞方面，LLM 单独使用时并非理想方案 [21] ——“……在复杂和高层次任务（如新漏洞识别）上，其效果不如现有工具。”** 与此相呼应，先前的研究也表明，LLMs 的能力主要体现在**简单任务或小规模程序**上 [1, 25, 26]。其原因在于，LLMs 仍存在多方面不足。首先，它们易受**`幻觉（hallucination）`**问题困扰 [11]：在分析错误代码时，模型可能**捏造不存在的事实**，试图解释代码作者的意图，而非真正识别漏洞 [17, 31]。其次，LLMs 具有**`随机性（stochasticity）`**，可能产生不一致或完全错误的结果，从而进一步干扰漏洞检测流程 [41]。最后，LLMs 的**`上下文窗口有限`**，只能同时处理相对较小规模的代码片段，这也限制了其在大型项目中的适用性。

​	对此，我们提出 **LLift**，一个在分析 UBI（Use-Before-Initialization）漏洞中将静态分析与大语言模型（LLM）相结合的**全自动化框架**。我们的方案包含若干创新组件：首先，LLift 执行**后约束引导的路径分析（post-constraint guided path analysis）**，用于验证变量“使用”时是否已初始化的路径可行性——这是静态分析和符号执行都难以解决的任务。其次，为了高效地与 LLM 交互，我们采用**任务分解（task decomposition）**策略，将分析过程拆解为多个步骤而非一次性完成。第三，我们采用**渐进式提示（progressive prompting）**，即仅在必要时逐步提供信息，而不是一次性输入大规模代码范围。最后，我们提出**自验证机制（self-validation）**，通过在不同阶段请求 LLM 自行复查其回答，以获得更准确、可靠的结果。

​	我们实现了 LLift 的原型并在真实场景中进行了测试。针对 UBITect 因时间或内存限制而分析未果的案例，LLift 成功识别出了 **13 个此前未知的 Linux 内核 UBI 漏洞**，并已由 Linux 社区确认。在近 1000 个案例中，LLift 共产生 26 个真实漏洞报告，**精准率达到 50%**。此外，我们还在 UBITect 先前发现的所有已知漏洞上对 LLift 进行了验证，**召回率达到 100%**。

​	我们总结的贡献如下：

- **新机会（New Opportunities）**：我们提出了一种新颖的静态分析方法，通过利用大语言模型（LLMs）的能力同时提高静态分析的**精确性与可扩展性**。据我们所知，这是首个在大规模、真实世界数据集上使用 LLM 辅助静态分析进行缺陷发现的工作。
- **新方法（New Methodologies）**：我们开发了 LLift，一个创新且全自动化的代理框架，将静态分析与 LLM 有机结合。LLift 采用多种提示策略来与 LLM 交互，从而引导模型给出更准确、可靠的回答。
- **结果（Results）**：我们对近 1000 个案例进行了详尽评估，取得了较高的精确率（50%）与召回率（100%）。此外，我们的工作还发现了 13 个此前未知的漏洞。
- **开源（Open source）**：为推动该研究方向的发展，我们将公开发布全部代码与数据，促进 LLM 辅助程序分析的新研究探索。
