# Blended Analysis for Predictive Execution



## abstract

​	**尽管大型语言模型（LLMs）在理解源代码和描述性文本方面表现出色，但它们在推理动态程序行为（例如执行轨迹、代码覆盖率预测以及运行时错误预测）方面仍存在局限性**——这些任务通常需要实际执行程序。

​	为提升 LLM 在动态行为预测中的能力，我们结合 **程序分析（Program Analysis, PA）** 与 **LLM** 的优势，提出了 **PredEx**，一种用于 Python 的预测执行器（predictive executor）。我们的核心思想是：通过 **PA 与 LLM 的融合分析（blended analysis）**，**<u>*利用程序分析来引导 LLM 预测程序的执行轨迹。我们将预测执行任务分解为多个更小的子任务，并在执行顺序可以确定时，充分利用其确定性。当执行顺序存在不确定性时，我们采用 基于变量的预测性反向切片（predictive backward slicing per variable），即仅保留先前执行轨迹中对每个变量产生影响的部分，从而将取值预测问题分解为若干显著更简单的子问题*</u>。**

​	在real-world数据集上的实证评估表明，PredEx 在完整**执行轨迹预测**上的准确率相较最新模型提升了 **31.5%–47.1%**；在正确预测执行轨迹前缀的数量上提升了 **8.6%–53.7%**。在预测下一个执行语句的任务中，PredEx 相较基线模型的相对提升幅度达到 **15.7%–102.3%**。最后，我们展示了 PredEx 在两个任务中的实用性：**静态代码覆盖率分析** 和 **对（不）完整代码的运行时错误静态预测**。

​	



## Intro

​	大型语言模型（Large Language Models, LLMs）[6, 15] 或预训练语言模型（Pre-trained Language Models, PLMs）[11, 14] 已在自然语言处理任务中展现出卓越的能力，包括代码生成、摘要生成、问答等。然而，尽管具备这些优势，它们在**推理动态程序行为**方面依然存在困难，例如**执行轨迹预测** [7, 14]。**<u>`这种局限性的根源在于：程序的执行轨迹依赖于运行时上下文（runtime context）、变量状态（variable state）以及程序内部复杂的控制流（control flow）——而这些信息并未直接编码在模型所学习的静态表示中。`</u>**现有的 LLM 主要从**静态代码（static code）\**中学习，而静态代码缺乏进行精确动态分析所需的\**执行关系（execution relations）**。**因此，当输入发生细微变化时，模型往往无法正确预测程序行为的变化**。这一现象凸显出开发**专门化模型**或**混合式方法（hybrid approaches）**的必要性，以结合 LLM 的优势与更适合动态分析的工具，从而更有效地建模和预测程序的运行行为。

​	我们的研究旨在**提升 LLM/PLM** 在**`无需实际执行`的情况下预测程序运行时执行轨迹**的能力，我们将这种能力称为 **预测执行（predictive execution）** 或 **静态代码执行（static code execution）**。`这种预测能力能够在软件测试与质量控制中发挥重要作用，因为它可以帮助识别程序中哪些部分可能会被执行、哪些部分可能被忽略。`此外，它使得在**无需真实运行程序**的情况下，对一系列**动态行为相关分析**成为可能，例如：动态切片（dynamic slicing），动态依赖分析（dynamic dependence analysis），测试用例覆盖率分析（test case coverage analysis），运行时错误检测（runtime error detection）等。

​	值得注意的是，**预测执行（predictive execution）** 在**无需实际执行的情况下静态`检测源代码中的运行时错误`**方面具有重要潜力。虽然像 **StackOverflow（S/O）** 这样的在线论坛为开发者提供了极为丰富的知识资源，但其内容也可能对采用这些代码片段的应用带来显著风险 [9, 13, 19]。因此，**及早检测这些含有缺陷或漏洞的在线代码片段**，对于避免引入安全隐患与减少无效适配工作具有关键意义。然而，由于讨论帖的上下文特性，这些在线代码片段通常是**不完整的**：它们可能**缺少必要的 import 语句（无论是内置库还是外部库）**，也可能缺少**外围方法或类的定义**，从而使得这些代码**无法直接执行** [27]。<u>传统的**静态分析（static analysis）**与**符号执行（symbolic execution)**技术  在此类场景中表现不足，因为它们依赖于**完整的程序代码与依赖库**</u>。因此，**面向不完整代码的预测执行**成为一种极具吸引力的替代方案。这种方法允许在静态代码执行分析中容忍一定程度的不精确性，从而使模型能够**近似地模拟代码的动态行为**，并实现对潜在运行时错误的预测性检测。

​	

​	对**预测执行（predictive execution）**的研究方法大体可以分为两类。**第一类**遵循**预训练语言模型（Pre-trained Language Model, PLM）范式**。例如，**CodeExecutor** [14] 需要训练数据中同时包含源代码、输入值、完整的执行轨迹，以及每个执行步骤中的变量取值。由于预测错误的累积效应（cascading errors），当变量值预测不准确时，模型在分支与循环中的预测行为也会随之出错。**TRACED** [8] 提出了一种基于执行数据的预训练策略，但在条件判断与循环结构的预测方面仍存在困难。它完全依赖于循环中**最后一行执行的结果**来推断程序状态，并通过变量取值范围近似表示执行行为。**第二类**方法则利用 **LLM 的代码理解能力**。**Tufano 等人** [22] 基于 **GPT-4** 的代码推理能力，用于预测代码覆盖率。然而，由于 LLM 的**幻觉现象（hallucination）**及**错误传播（error propagation）**问题，该方法在处理复杂结构时表现受限。**CodePilot** [7] 借助 \**Chain-of-Thought (CoT) 规划\** [24] 来引导 LLM 自主生成代码覆盖的执行计划。**NExT** [16] 则通过 CoT 机制训练 LLM 观察执行轨迹（即在执行语句处标注变量状态）。然而，这两种基于 CoT 的方法在预测执行过程中均存在**违反程序语义（program semantics violation）**的现象。



> 从理论层面上讲，**LLM（包括CoT等推理增强方法）确实仍然是基于统计关联的语言建模**，而不是在“真正地执行”程序语义。

​	在本文中，为了提升 LLM 和 PLM 在预测执行（predictive execution）方面的能力，我们提出了 **PredEx** —— 一种 **融合分析（blended analysis）** 方法，用于结合 **程序分析（Program Analysis, PA）** 与 **LLM/PLM** 的优势，在给定代码片段及其输入的情况下，预测下一条执行语句以及完整执行轨迹。在讨论预测执行时，我们将 LLM 与 PLM 统称为 **LLM**，除非在特定情境下需要区分二者。

​	**`在 PredEx 中，我们假设：通过为 LLM 提供来自 PA 的支持，并向其输入相关语句，可以引导 LLM 对**一小部分语句的执行**进行推理，而无需实际执行程序。`**我们将静态代码执行任务拆分为多个子任务，并根据各自优势将每个子任务分配给 PA 或 LLM。**粗粒度层面**：将预测执行问题划分为**每一步预测下一条执行语句**。当 PA 可以做出**确定性决策**时，例如在一个代码块内确定下一条执行语句，我们优先使用 PA，从而降低 LLM 潜在的不准确性。**细粒度层面**：当 PA 无法确定执行流，尤其是在**分支点**（如 if、while、for 等）时，我们让 LLM 处理对分支方向决策至关重要的任务。在 LLM 处理分支决策的过程中，我们利用 PA 协助 LLM 聚焦于**与分支决策相关的关键语句**，以提高推理的准确性和效率。



​	具体而言，在分支点处，LLM 的预测执行任务可以划分为两个子任务：**值评估器（value evaluator）\**和\**条件评估器（condition evaluator）**。**值评估器（Value Evaluator）**
 值评估器的任务是确定参与分支决策的变量的取值。通过从这些变量出发进行**反向切片（backward slicing）**，我们可以帮助 LLM 仅关注**对变量取值有实际影响的相关语句**。我们构建反向切片，并结合当前预测的语句和轨迹中的路径，近似得到动态反向切片，这里称之为 **预测性反向切片（predictive backward slice）**。通过这些切片和预测的变量值，LLM 可以识别代码中影响当前变量取值的模式和依赖关系。我们的核心思想是：**按变量进行反向切片**（即将先前执行轨迹切分为仅影响每个变量的部分），可以将复杂的变量取值预测问题分解为多个更简单的子问题。此外，通过对 LLM 进行少样本学习（few-shot learning）或针对外部 API 调用进行微调（fine-tuning），我们使其能够推导出这些调用的输出值。

​	**条件评估器（Condition Evaluator）** 条件评估器基于值评估器获得的变量值进行工作。在分支点给定变量的取值后，条件评估器可以预测条件表达式的结果。在实现条件评估时，可以采用多种策略：第一，像值评估器一样直接使用 LLM 进行预测；第二，利用编译器中的表达式求值器（expression evaluator）进行计算；第三，构建概率条件评估器（probabilistic condition evaluator）来估计条件成立的概率；最后，可以结合多种策略以获得更稳健的预测结果。



​	我们将 **PredEx** 构建为 Python 的解释器，包括两种实现方式：

1. **PA + CodeExecutor**（程序分析 + 预训练语言模型 CodeExecutor）
2. **PA + GPT-4**（程序分析 + GPT-4）



​	对于**完整代码片段**，在 PA 支持下，PredEx 相较于单独使用 CodeExecutor 和 GPT-4，在预测完整执行轨迹的准确率上分别提升了 **31.5%** 和 **47.1%**。在生成执行轨迹前缀的正确率方面，PredEx 比未使用 PA 的 CodeExecutor 和 GPT-4 高 **8.6%–53.7%**。在预测下一条执行语句时，其相对提升范围为 **5.7%–102.3%**。对于**不完整代码片段**，在 PA 支持下，PredEx 相较于 CodeExecutor 和 GPT-4，在执行轨迹预测的准确率上分别提升 **36%** 和 **41.2%**。

​	我们还展示了 PredEx 在两个任务中的有效性：**静态代码覆盖率计算**以及**运行时错误的静态预测**。PredEx（PA + GPT-4）能够将 GPT-4 的覆盖率预测准确率提升 **33.5%**（语句覆盖率）和 **26.2%**（分支覆盖率）。在不完整代码的运行时错误预测任务中，PredEx（PA + GPT-4）也能达到 **89.5%** 的高准确率。

​	简而言之，主要贡献包括：
 a) **预测执行（Predictive Execution）**：PredEx 是一款无需实际执行即可预测 Python 代码及其输入值执行轨迹的解释器，适用于完整或不完整代码。
 b) **融合分析框架（Blended Analysis Framework）**：提出了程序分析与 LLM 结合的融合分析方法，用于预测执行，其性能优于单独方法。其他静态/动态分析方法也可以以类似方式与 LLM 融合。
 c) **预测执行的应用（Applications of Predictive Execution）**：展示了 PredEx 在静态代码覆盖率预测以及不完整代码的运行时错误静态预测中的实用性。
 d) **实证评估（Empirical Evaluation）**：实验结果表明，PredEx 在有效性及下游任务上均优于现有最先进基线方法。
